# AI 行业温度计 WebApp（内网版）PRD v0.1

> 文档目的：用于指导 AI/研发团队在公司服务器/内网环境中，从 0 开发一个可长期维护的数据化 WebApp  
> 范围说明：本 PRD 覆盖 v0 版本（MVP）必做功能，并为未来版本（鉴权、筛选聚合等）做结构预留  
> 关键约束：初期不做登录鉴权；数据量小；强调易部署；需要数据库固化与可持续维护  

---

## 1. 背景与目标

### 1.1 背景
团队希望以“二维矩阵（类似表格）”的方式，直观呈现不同“游戏类型”与不同“游戏行业工种”交叉点下，当前可用的 SOTA AI 工具是什么，以及其成熟度/可用性程度。

### 1.2 产品目标
- 以 **表格化矩阵** 展示：  
  - 横轴：一系列“游戏类型”卡片  
  - 纵轴：一系列“游戏行业工种”（两级层级）  
  - 单元格：该交叉点的 **AI 工具卡片**（若有），并用 **颜色编码** 表示成熟度  
- 支持长期维护：可在 Web 端完成增删改查与排序，数据落库固化
- 内网部署：易部署、易备份、低维护成本

### 1.3 非目标（v0 不做）
- 不做用户登录鉴权/权限体系（但需为未来接入预留字段与扩展点）
- 不做导出/分享/订阅/通知等扩展能力
- 不做复杂的上传限制（格式/尺寸/大小等），仅前端提示并统一裁剪展示

---

## 2. 术语与概念

- **游戏类型（Game Type）**：位于矩阵横轴的“类型卡片”。名称可为多个维度组合后的最终命名（例如“2.5D 俯视角 RPG”）。
- **游戏行业工种（Role）**：位于矩阵纵轴。具有 **两级层级**（一级大类 + 二级子类）。
- **单元格（Cell）**：一个 Game Type × 一个 Role（通常是二级 Role）的交叉点。
- **工具卡片（Tool Card）**：单元格中的信息载体，包含：成熟度等级 + 工具信息（名称、官网链接、描述、报告链接）。
- **成熟度等级（Maturity Level）**：离散档位 **5 档**（不含灰色）。  
- **灰色状态（Gray / Missing）**：用于表示该单元格“缺失/暂时不存在/无工具”。v0 简化为 **统一缺失语义**（不区分不适用/未评估/暂无工具）。

## 3. 用户与使用场景

### 3.1 用户
- 内网用户（默认所有访问者均可使用管理功能）
- v0 无鉴权：任何访问者都可增删改查与排序

### 3.2 核心场景
1. 维护者新增一个游戏类型，并上传多张示例海报
2. 维护者新增/调整工种大类与子类，并拖拽排序
3. 维护者在某个单元格添加一个工具卡片，选择成熟度等级并填写工具信息
4. 浏览者浏览矩阵，通过颜色快速判断“AI 结合程度高/低”，点击工具卡片查看详情
5. 维护者长期更新：替换某工具为新 SOTA，调整成熟度等级或更新描述/链接

## 4. 功能需求（v0）

### 4.1 矩阵总览页（核心页面）

#### 4.1.1 页面结构
- 顶部横轴：游戏类型卡片列表（可水平滚动）
- 左侧纵轴：工种两级列表（垂直）
  - 一级：工种大类（组标题）
  - 二级：工种子类（行项目，参与与游戏类型的交叉）
- 主体区域：矩阵单元格网格
  - 每个单元格最多展示 **0 或 1** 个工具卡片（v0 固定为 0/1，不支持多个工具）

#### 4.1.2 单元格展示规则
- 若该单元格有工具卡片：
  - 显示成熟度颜色块（红绿灯风格）
  - 显示工具名称
- 若没有工具卡片：
  - 显示灰色（缺失/暂时不存在/无工具）占位
  - v0：灰色语义不区分原因（不适用/未评估/暂无工具等后续再扩展）

#### 4.1.3 工具卡片点击交互
- 点击工具卡片后展示“工具详情”
  - 形式可为：弹窗（Modal）或新页面（Detail Page）
  - v0 可任选其一，但需保持交互清晰、信息完整

---

### 4.2 游戏类型管理（横轴）

#### 4.2.1 CRUD
- 添加游戏类型
- 编辑游戏类型
- 删除游戏类型（建议默认软删除以防误删，见数据设计）

#### 4.2.2 拖拽排序
- 支持在横轴上对游戏类型卡片进行拖拽排序
- 排序结果需持久化到数据库

#### 4.2.3 多维组合结构（结构化存储 + 可配置维度体系）
> 用户当前只关心最终名称，但希望从 v0 起将游戏类型以更结构化方式存储，并且未来支持筛选/聚合/管理。  
> 同时，未来在新增/编辑游戏类型时，希望可以对“每个维度”进行选择，但维度体系尚未定。

v0 要求（必须）：
- 每个游戏类型必须有：
  - 最终名称（必填）
  - 结构化维度组合信息（可为空或部分填写，但需有结构化字段承载）
- 维度体系本身需可配置（C）：维度名、维度取值均可维护

v0 交互建议（落地可行且利于后续维护）：
- 在“新增/编辑游戏类型”时：
  - 必填：最终名称
  - 可选：选择若干维度-取值（从可配置维度体系中选择）
  - 允许未配置维度体系时，先只填最终名称；后续再补全维度关联

#### 4.2.4 示例海报管理
- 每个游戏类型可关联 **多张** 示例海报
- 支持上传/删除海报
- v0 不强制限制格式/尺寸/大小（通过前端文案提示即可）
- 前端展示需统一裁剪到适合展示的尺寸（例如固定比例缩略图）

---

### 4.3 工种管理（纵轴，两级层级）

#### 4.3.1 两级结构
- 工种必须是“两级”：
  - Level 1：工种大类（例如“2D美术”、“3D美术”、“动画与动捕”等）
  - Level 2：工种子类（例如“2D原画”、“3D建模”、“动作捕捉”等）
- 两级都支持：添加 / 编辑 / 删除

#### 4.3.2 拖拽排序（规则确认：A）
- 支持拖拽排序
- v0 排序规则（已确认 A）：
  - 子类只能在所属大类内拖拽排序
  - **不允许** 将子类拖拽到其他大类（不允许跨组改变归属）
- 大类之间也应可拖拽排序（建议支持）

---

### 4.4 单元格工具卡片管理（核心）

#### 4.4.1 0/1 卡片规则（已确认 A）
- 每个单元格最多存在 **1** 个工具卡片
- 支持：
  - 添加工具卡片（当单元格为空时）
  - 编辑工具卡片（当单元格已有卡片时）
  - 移除工具卡片（删除该单元格卡片，变为灰色缺失）

#### 4.4.2 成熟度等级与颜色编码
目标：遵循“红绿灯”风格，表示 AI 结合程度高/低，并支持中间档位。

已确认：
- 成熟度等级使用 **离散档位 5 档**（不含灰色）
- 灰色为缺失/暂时不存在（v0 统一语义）

v0 建议档位（可在配置中维护，默认如下）：
- Level 1：红（AI 结合程度低）
- Level 2：橙
- Level 3：黄
- Level 4：黄绿
- Level 5：绿（AI 结合程度高）
- Gray：缺失/无工具（单元格无卡片时显示）

实现约束（建议）：
- 数据库中存储 `maturity_level`（1-5）
- 颜色展示由前端映射（避免未来换色/换主题导致数据迁移）
- 若未来需要更灵活，也可做“等级配置表”，让 level->label->color 可维护（见数据设计）

#### 4.4.3 工具信息字段（已确认）
每个工具卡片需至少存储/展示：
- 工具名称（必填）
- 工具官网链接（必填）
- 工具简短描述（必填）
- 工具报告链接（可选）

#### 4.4.4 工具详情展示
点击工具卡片展示：
- 工具名称
- 成熟度等级（含颜色表现）
- 工具官网链接（可点击）
- 工具简短描述
- 工具报告链接（若有）
（可扩展字段后续版本再加）

### 4.5 未来鉴权接入预留（v0 必须预留字段）

虽然 v0 不做鉴权，但需在数据库中为未来“接入公司现有鉴权模块”预留审计字段。

v0 必须：
- 关键业务表需要包含：
  - `created_at` / `updated_at`

  - `created_by` / `updated_by` 可为空或填 `"system"`/`"anonymous"`（实现选一种即可）
- 后续接入鉴权时：
  - 写入真实的公司 user_id（由公司模块提供）

## 6. 数据设计（v0 建议方案）

### 6.1 数据库选型（易部署优先）
- 推荐：SQLite（极易部署，单文件，适合小数据量）
- 可替代：PostgreSQL（若公司内网已有通用 DB 基座与运维体系）

v0 选择建议：
- 若目标是“最省事部署”：SQLite
- 若目标是“更标准企业化/并发更稳”：Postgres

### 6.2 实体与关系概览
- GameType（游戏类型）
- RoleGroup（工种大类）
- Role（工种子类，归属某 RoleGroup）
- ToolCell（单元格：GameType × Role 的工具卡片记录，0/1）
- GameTypePoster（游戏类型海报，多张）
- Dimension / DimensionValue（维度体系配置）
- GameTypeDimensionValue（游戏类型与维度取值关联，多对多）
- （可选）MaturityLevelConfig（成熟度等级配置，若希望可维护 label/color）

### 6.3 表结构（字段建议）

#### 6.3.1 游戏类型表：`game_type`
- `id` (PK)
- `name` (TEXT, 必填) — 最终展示名称
- `sort_order` (INT, 必填) — 拖拽排序序号
- `created_at` (DATETIME)
- `updated_at` (DATETIME)
- `created_by` (TEXT)
- `updated_by` (TEXT)
- `is_deleted` (BOOL, 默认 false) 或 `deleted_at`（软删方案二选一）

#### 6.3.2 游戏类型海报表：`game_type_poster`
- `id` (PK)
- `game_type_id` (FK -> game_type.id)
- `file_path` (TEXT, 必填) — 文件存储路径或 URL
- `sort_order` (INT, 必填) — 海报展示顺序
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

> 存储策略建议（内网易部署）：文件保存到服务器本地目录（例如 `/data/uploads/posters/`），DB 存 `file_path`。

#### 6.3.3 工种大类表：`role_group`
- `id` (PK)
- `name` (TEXT, 必填)
- `sort_order` (INT, 必填)
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

#### 6.3.4 工种子类表：`role`
- `id` (PK)
- `role_group_id` (FK -> role_group.id, 必填)
- `name` (TEXT, 必填)
- `sort_order` (INT, 必填) — 在所属大类内排序
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

#### 6.3.5 单元格工具卡片表：`tool_cell`
> 代表 GameType × Role 的交叉点的唯一工具卡片（0/1）

- `id` (PK)
- `game_type_id` (FK -> game_type.id, 必填)
- `role_id` (FK -> role.id, 必填)
- `tool_name` (TEXT, 必填)
- `maturity_level` (INT, 必填, 1-5)
- `official_url` (TEXT, 必填)
- `short_desc` (TEXT, 必填)
- `report_url` (TEXT, 可选)
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

唯一性约束（关键）：
- `UNIQUE(game_type_id, role_id)`  
确保一个单元格最多 1 条记录（满足你选择 A：0/1）

#### 6.3.6 维度体系表：`dimension`
- `id` (PK)
- `name` (TEXT, 必填) — 例如“玩法”、“画风”、“建模维度”、“视角”
- `sort_order` (INT, 必填)
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

#### 6.3.7 维度取值表：`dimension_value`
- `id` (PK)
- `dimension_id` (FK -> dimension.id, 必填)
- `value` (TEXT, 必填) — 例如“动作”、“像素”、“2.5D俯视角”
- `sort_order` (INT, 必填)
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

#### 6.3.8 游戏类型-维度取值关联表：`game_type_dimension_value`
- `id` (PK)
- `game_type_id` (FK -> game_type.id, 必填)
- `dimension_value_id` (FK -> dimension_value.id, 必填)
- `created_at` / `updated_at` / `created_by` / `updated_by`
- `is_deleted` / `deleted_at`

唯一性约束（建议）：
- `UNIQUE(game_type_id, dimension_value_id)`

> 说明：你目前只关心最终 name，但选择 C（维度体系可配置）。  
> 这个设计允许：  
> - 维度体系尚未定义完整时，游戏类型照样能创建（只填 name）  
> - 未来补充维度后，可给已有游戏类型补挂维度取值，实现筛选/聚合

#### 6.3.9（可选）成熟度等级配置表：`maturity_config`
> 若你希望将 1-5 的 label/说明/颜色也做成可维护配置

- `level` (PK, INT, 1-5)
- `label` (TEXT) — 可写“低/较低/中/较高/高”或你自定义文案
- `color_key` (TEXT) — 前端映射用 key（不建议存 HEX）
- `sort_order` (INT)

v0 可不做该表，直接前端写死映射即可。

## 7. 接口需求（API，v0）

> 仅定义能力范围，具体路径可由技术栈决定（REST/GraphQL 均可）。  
> v0 无鉴权，但应为未来鉴权留出中间件位置。

### 7.1 GameType
- `GET /game-types` 获取列表（按 sort_order）
- `POST /game-types` 新增
- `PUT /game-types/{id}` 编辑
- `DELETE /game-types/{id}` 删除（软删）

- `POST /game-types/{id}/posters` 上传海报（可多张）
- `DELETE /posters/{poster_id}` 删除海报
- `PUT /posters/{poster_id}` 更新海报排序（或批量更新）

- `PUT /game-types/sort` 批量更新排序（拖拽结果持久化）

### 7.2 Role（两级）
- `GET /role-groups` 获取大类列表（含子类，按 sort_order）
- `POST /role-groups` 新增大类
- `PUT /role-groups/{id}` 编辑大类
- `DELETE /role-groups/{id}` 删除大类（软删）

- `POST /roles` 新增子类（需 role_group_id）
- `PUT /roles/{id}` 编辑子类
- `DELETE /roles/{id}` 删除子类（软删）

- `PUT /role-groups/sort` 大类批量排序
- `PUT /roles/sort` 子类批量排序（仅组内）

### 7.3 ToolCell（单元格工具卡片）
- `GET /tool-cells` 获取全部工具卡片（用于矩阵渲染）
- `POST /tool-cells` 新增卡片（需 game_type_id + role_id，且满足唯一约束）
- `PUT /tool-cells/{id}` 编辑卡片
- `DELETE /tool-cells/{id}` 移除卡片（软删）

### 7.4 Dimension（维度体系，可配置 C）
- `GET /dimensions` 获取维度列表（含取值）
- `POST /dimensions` 新增维度
- `PUT /dimensions/{id}` 编辑维度
- `DELETE /dimensions/{id}` 删除维度

- `POST /dimension-values` 新增维度取值
- `PUT /dimension-values/{id}` 编辑维度取值
- `DELETE /dimension-values/{id}` 删除维度取值

- `PUT /dimensions/sort` 维度排序
- `PUT /dimension-values/sort` 取值排序（按所属维度）

- `PUT /game-types/{id}/dimension-values` 设置某游戏类型选择的取值（覆盖式或增量式，技术选一种）

## 8. 前端交互与编辑入口（v0）

### 8.1 总览页交互
- 横轴游戏类型卡片：
  - 支持拖拽排序（拖拽结束自动保存）
  - 点击可查看该类型信息（可选）或进入编辑
- 纵轴工种：
  - 大类可折叠/展开（建议）
  - 子类行与横轴交叉形成单元格
- 单元格：
  - 空：显示灰色占位 + “添加工具”入口（仅管理者逻辑，v0 全员可见）
  - 有卡片：显示颜色等级 + 工具名称
  - 点击卡片：打开工具详情
  - 卡片上提供编辑/删除入口（hover 或右键菜单均可）

### 8.2 游戏类型编辑
- 字段：
  - 最终名称（必填）
  - 维度取值选择（可选，多选；从“维度体系”中选择）
  - 海报管理：上传/删除/排序
- 说明：
  - 若维度体系尚未配置完成，可先只填最终名称
  - 后续可再补维度取值

### 8.3 工种编辑
- 大类管理：增删改、排序
- 子类管理：在大类内增删改、组内排序（不跨组）

### 8.4 维度体系管理（C）
- 维度（dimension）列表：增删改、排序
- 维度取值（dimension_value）：在某维度下增删改、排序

## 9. 文件/图片存储方案（v0）

### 9.1 方案目标
- 易部署
- 易备份
- 可删除与可追踪

### 9.2 推荐方案（默认）

## 10. 数据一致性与边界规则

### 10.1 唯一性
- `tool_cell` 必须 `UNIQUE(game_type_id, role_id)`，保证单元格 0/1
- `game_type_dimension_value` 建议唯一约束防重复

### 10.2 删除行为（建议软删）
- 删除游戏类型：
  - 其关联的 `tool_cell` 与 `poster` 与 `dimension` 关联数据的处理策略：
    - v0 建议：软删主对象，相关数据保留或联动软删（实现可选，但需明确）
- 删除工种子类：
  - 其关联 `tool_cell` 需同步处理（软删或阻止删除并提示依赖）
- v0 建议统一：软删 + 在 UI 提示“有关联数据，删除会同时移除关联卡片”

### 10.3 灰色状态语义（已确认 A）
- v0：单元格无卡片 => 灰色缺失
- 不区分“不适用/未评估/暂无工具”

## 15. v0 需求确认结论（来自本次对话，确保不遗漏）

- 部署：公司服务器/内网 WebApp
- 页面：表格矩阵
  - 横轴：游戏类型卡片（可增删改、拖拽排序）
  - 纵轴：游戏行业工种（两级层级，均可增删改；拖拽排序；子类仅组内排序）
  - 单元格：0/1 工具卡片
- 工具卡片：
  - 展示：颜色编码 + 工具名称
  - 点击：查看工具详情
  - 可增删改：颜色等级与工具信息
- 颜色编码：
  - 红绿灯风格
  - 成熟度等级：离散 5 档（红/橙/黄/黄绿/绿），灰色表示缺失
  - v0 灰色语义：统一缺失（不区分原因）
- 工具字段：
  - 官网链接（必填）
  - 简短描述（必填）
  - 报告链接（可选）
- 海报：
  - 每个游戏类型可有多张示例海报
  - 支持上传/删除
  - v0 不加限制，前端提示并统一裁剪成展示大小
- 游戏类型多维组合：
  - 当前只关心最终名字
  - 但希望从 v0 起结构化存储并为未来筛选/聚合预留
  - 选择 C：维度体系本身可配置（维度与取值可维护）
- 数据库：
  - 数据需固化落库
  - 小数据量，优先易部署（推荐 SQLite）
- 鉴权：
  - v0 不做
  - 但数据库必须预留审计字段：created_by/updated_by（cuser/uuser）与 created_at/updated_at

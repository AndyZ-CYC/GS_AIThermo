# 00 项目总览 — AI 行业温度计 WebApp（内网版）

> 本文档为项目背景与全局上下文，供所有后续模块 PRD（01–09）的 AI Agent 阅读时建立共同认知基础。  
> 版本：v0（MVP）  
> 日期：2026-03

---

## 0.1 项目背景

团队希望以**二维矩阵（类似表格）**的方式，直观呈现不同"游戏类型"与不同"游戏行业工种"交叉点下，当前可用的 SOTA AI 工具是什么，以及其成熟度/可用性程度。

该 WebApp 部署在公司内网服务器，面向内网用户。

---

## 0.2 产品目标

- 以**表格化矩阵**展示：
  - **横轴**：一系列"游戏类型"卡片（可拖拽排序，可附多张示例海报）
  - **纵轴**：一系列"游戏行业工种"（两级层级：大类 + 子类）
  - **单元格**：该交叉点的 AI 工具卡片（若有），并用**颜色编码**表示成熟度
- 支持**长期维护**：可在 Web 端完成增删改查与排序，数据落库固化
- **内网部署**：易部署、易备份、低维护成本

---

## 0.3 非目标（v0 不做）

- 不做用户登录鉴权/权限体系（但**必须**为未来接入预留字段与扩展点，见 09 文档）
- 不做导出/分享/订阅/通知等扩展能力
- 不做复杂的上传限制（格式/尺寸/大小等），仅前端提示并统一裁剪展示
- 不做搜索/筛选功能（后续版本扩展）
- 不做版本化/历史记录

## 0.4 术语与核心概念

| 术语 | 定义 |
|------|------|
| **游戏类型（Game Type）** | 矩阵横轴上的"类型卡片"。名称可为多维度组合后的最终命名（例如"2.5D 俯视角 RPG"）。每个游戏类型可关联多张示例海报。 |
| **游戏行业工种（Role）** | 矩阵纵轴上的条目。具有**两级层级**：Level 1（工种大类）和 Level 2（工种子类）。 |
| **工种大类（RoleGroup）** | 纵轴一级分组，例如"2D 美术"、"3D 美术"。本身不直接参与交叉，是 Role 的容器。 |
| **工种子类（Role）** | 纵轴二级条目，例如"2D 原画"、"3D 建模"。参与与游戏类型的交叉，形成单元格。 |
| **单元格（Cell）** | 一个 Game Type × 一个 Role（二级）的交叉点。v0 每个单元格最多存在 **0 或 1** 个工具卡片。 |
| **工具卡片（Tool Card）** | 单元格中的信息载体。包含：成熟度等级 + 工具名称 + 官网链接 + 简短描述 + 报告链接（可选）。 |
| **成熟度等级（Maturity Level）** | 离散 5 档（1=红，2=橙，3=黄，4=黄绿，5=绿），表示该交叉点的 AI 结合程度高低。 |
| **灰色状态（Gray / Missing）** | 单元格无工具卡片时显示的状态。v0 统一语义：缺失/暂时不存在，不区分具体原因。 |
| **示例海报（Poster）** | 游戏类型关联的参考截图/海报图片。多张并排展示，数量过多时允许图片重叠。 |
| **维度体系（Dimension System）** | 可配置的游戏类型分类体系（维度名 + 维度取值）。v0 起结构化存储，供未来筛选/聚合使用。 |

## 0.5 整体数据实体关系（概览）

```
Dimension (维度)
    └── DimensionValue (维度取值，多个)

GameType (游戏类型)
    ├── GameTypePoster (示例海报，多张)
    └── GameTypeDimensionValue (关联维度取值，多对多)

RoleGroup (工种大类)
    └── Role (工种子类，多个)

ToolCell (单元格工具卡片)
    ├── FK -> GameType
    └── FK -> Role
    UNIQUE(game_type_id, role_id)  -- 保证每单元格最多1张
```

## 0.7 删除行为全局规则（重要）

**v0 统一原则：存在级联依赖时，阻止删除并提示用户先清理关联数据。**

| 删除对象 | 存在依赖时的行为 |
|----------|----------------|
| 游戏类型（GameType） | 若存在关联的 ToolCell 或 Poster，**阻止删除**，提示先删除关联卡片和海报 |
| 工种大类（RoleGroup） | 若存在子工种（Role），**阻止删除**，提示先删除子工种 |
| 工种子类（Role） | 若存在关联的 ToolCell，**阻止删除**，提示先删除关联工具卡片 |
| 维度（Dimension） | 若存在维度取值（DimensionValue），**阻止删除**，提示先删除取值 |
| 维度取值（DimensionValue） | 若已被某游戏类型关联，**阻止删除**，提示先解除关联 |

所有删除操作均使用**软删除**（`is_deleted = true` 或设置 `deleted_at`），不进行物理删除。

## 0.8 成熟度等级颜色编码（全局）

| 等级值 | 语义 | 颜色 |
|--------|------|------|
| 1 | AI 结合程度低 | 红（Red） |
| 2 | 较低 | 橙（Orange） |
| 3 | 中 | 黄（Yellow） |
| 4 | 较高 | 黄绿（Yellow-Green） |
| 5 | AI 结合程度高 | 绿（Green） |
| — | 缺失/无工具 | 灰（Gray） |

- 颜色映射由**前端实现**（数据库只存 1-5 整数），避免换色时需要数据迁移。

## 0.9 审计字段规范（全局）

所有核心业务表**必须**包含以下字段，供未来鉴权接入使用：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 最后更新时间 |
| `created_by` | TEXT | 创建者（v0 填写 `"system"` 或置空） |
| `updated_by` | TEXT | 更新者（v0 填写 `"system"` 或置空） |

涉及软删除的表还需包含：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `is_deleted` | BOOL | 软删除标记，默认 false |
| `deleted_at` | DATETIME | 软删除时间，未删除时为 NULL |

## 0.10 技术约束

| 约束项 | 说明 |
|--------|------|
| 部署环境 | 公司服务器 / 内网，不联外网 |
| 数据库 | 优先 SQLite（单文件，极易部署）；可替代 PostgreSQL |
| 文件存储 | 服务器本地目录（例如 `/data/uploads/posters/`），DB 存路径 |
| 鉴权 | v0 无鉴权，任何内网用户均可增删改查 |
| 数据量 | 小（内网工具管理场景），无高并发要求 |

## 0.12 后续版本开放问题（v0 不做）

1. 灰色语义是否细分为"不适用 / 未评估 / 暂无工具"
2. 单元格是否支持多个工具（Top-N + 主推荐）
3. 工具信息是否需要更多字段（定价、平台、是否本地部署、使用门槛等）
4. 是否需要搜索/筛选（基于维度、工种、成熟度、工具名）
5. 是否需要版本化/历史记录（SOTA 变更追踪）
6. 鉴权接入（接入公司现有用户体系）
